01 Ontology and methodology

Typed medical meaning, then predefined practice.

The ontology gives every safety-relevant thing a type. The methodology tells teams how to apply those types through rigor profiles, ISO 42010 viewpoints, rules, workflow stages, review gates, and project bindings.

github.com/memoarchitect/memo
Ontology

Typed medical meaning

A minimal, medically grounded SysML v2 vocabulary published in one memo:: namespace for intended use, hazards, harms, risk controls, requirements, architecture, behavior, verification, and evidence.

Semantic links

Traceability as data

Layers do not connect by free-form arrows. Every link is a SemanticLink subtype with named roles and a status, so the linker can check the element kind on each end.

Methodology

Predefined practice

Teams start from a working structure rather than an empty model: pick rigor, pick viewpoints, apply rules, bind the method to a project, and extend through profiles.

02 Specification contents

One source of truth for typed safety evidence.

One namespace

A core memo:: namespace defines the vocabulary so viewpoints, methodology, examples, and compliance mappings can build on it without forking it.

Closure rules

Rules can check whether high-severity controls reach allocated behavior, acceptance criteria, verification cases, and evidence nodes.

Method profiles

Lightweight, balanced, or formal rigor profiles select viewpoints, required content, workflow stages, and review-gate exit criteria.

Extension points

Projects add packages and methodology profiles for alarms, cybersecurity, usability, interoperability, data messaging, configuration, or device-specific patterns.
